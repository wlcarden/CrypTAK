# Meshtastic ATAK Plugin -- Application-Layer Encryption

## Overview

The app-layer encryption feature adds **AES-256-GCM encryption** to CoT (Cursor on Target) payloads *before* they are handed to the Meshtastic radio. This creates a second layer of encryption independent of whatever the radio firmware provides.

### What It Protects

- Message content (PLI, chat, generic CoT) is encrypted end-to-end between ATAK devices
- Physical compromise of a LoRa radio does not expose message content
- Over-the-air interception of Meshtastic traffic reveals only ciphertext
- With epoch rotation enabled, compromise of the current key does not reveal past traffic (forward secrecy)

### What It Does NOT Protect

- **Traffic analysis** -- an observer can still see that messages are being sent, their timing, and approximate size
- **Meshtastic-level metadata** -- node IDs, hop counts, and radio-layer headers remain in the clear
- **Device compromise** -- if the Android phone running ATAK is compromised, the key can be extracted from memory or preferences
- **Denial of service** -- an attacker can jam the radio channel regardless of encryption
- **Sender authenticity** -- any device with the PSK can impersonate any callsign; there is no per-device identity binding
- **Replay attacks** -- captured encrypted packets can be retransmitted and will decrypt normally; GCM provides authentication but not replay protection
- **Audio** -- Codec2 voice memos are not covered by app-layer encryption

## Generating a Pre-Shared Key (PSK)

Generate a cryptographically random 256-bit key:

```bash
openssl rand -base64 32
```

This produces a 44-character base64 string. **Do not use the example key shown in this document or in the source code.** Always generate a fresh key for each deployment.

**Requirements:**

- Use a full-entropy 256-bit key generated by a cryptographic random number generator
- Do NOT use a passphrase, dictionary words, or any human-memorable string as the PSK
- Do NOT reuse PSKs across unrelated teams or deployments
- Do NOT transmit the PSK over the Meshtastic mesh itself
- Do NOT store the PSK in plaintext on shared drives or unencrypted channels

## Distributing the Key

Recommended distribution methods (in order of preference):

1. **QR code in person** -- Generate a QR code containing the base64 key string. Each team member scans it with ATAK or a QR scanner app. The `KeyImportReceiver` handles the import automatically.

2. **ATAK Data Package over TAK Server** -- Include the key in an encrypted ATAK data package distributed over a TLS-protected TAK Server connection.

3. **Secure side-channel** -- Send the key via Signal, encrypted email, or read it aloud over a secure voice channel.

**Never** distribute the key over the Meshtastic mesh, unencrypted SMS, or unencrypted email.

### Importing via QR Code

1. Generate the QR code from the base64 key string (any QR generator)
2. On the receiving device, scan the QR code with any scanner app
3. The plugin's `KeyImportReceiver` intercepts the scanned text, validates it is exactly 32 bytes when decoded, loads it into the encryption manager, and enables encryption

Alternatively, send the key programmatically via broadcast intent:

```java
Intent intent = new Intent("com.atakmap.android.meshtastic.IMPORT_ENCRYPTION_KEY");
intent.putExtra("key", base64KeyString);
context.sendBroadcast(intent);
```

### Manual Entry

1. Open ATAK Settings, then Tool Preferences, then Meshtastic Preferences
2. Scroll to **Application-Layer Encryption**
3. Enable **App-Layer Encryption**
4. Tap **Pre-Shared Key (PSK)** and paste the base64 key string
5. All team members must enter the identical key

## Key Derivation

When a PSK string is entered (manually or via QR import), the encryption key is derived as:

```
encryption_key = SHA-256(psk_string_bytes_utf8)
```

This means the key depends on the exact string entered, not the raw decoded bytes. All devices must enter the identical string to produce the same key.

## Epoch Rotation

Epoch rotation provides **forward secrecy**: if an attacker compromises the current key, they cannot decrypt messages encrypted under previous epochs.

### Mechanism

1. When epoch rotation is enabled, the encryption manager starts at **epoch 0** using the base key (SHA-256 hash of the PSK string)
2. After each epoch interval (default: 6 hours), the key advances using an HMAC-SHA256 chain:
   ```
   epoch_key[0] = base_key
   epoch_key[1] = HMAC-SHA256(epoch_key[0], "meshtastic-epoch-" || 0x00000001)
   epoch_key[2] = HMAC-SHA256(epoch_key[1], "meshtastic-epoch-" || 0x00000002)
   epoch_key[3] = HMAC-SHA256(epoch_key[2], "meshtastic-epoch-" || 0x00000003)
   ...
   ```
3. The derivation is **forward-only** -- knowing `epoch_key[n]` does not help derive `epoch_key[n-1]`
4. The epoch number is embedded in the V2 wire format so receivers know which key to use
5. Receivers can derive any epoch key from the base key by chaining forward, so a device that was offline for several epochs can still decrypt
6. Epoch numbers are bounded to a maximum of 8760 (one year at 1-hour intervals) to prevent denial-of-service from crafted packets

### Configuration

- **Enable Epoch Rotation**: Toggle in Meshtastic Preferences (requires encryption to be enabled first)
- **Epoch Rotation Interval**: 1h, 2h, 4h, 6h (default), 12h, or 24h
- **Current Epoch**: Read-only display showing the active epoch number

### Rotation Behavior

1. The encryption manager detects that the epoch has expired (based on elapsed time)
2. It saves the current key as the "previous epoch key"
3. It derives the new epoch key using HMAC-SHA256
4. New outbound messages use the new key with V2 wire format
5. For a brief window, the manager retains the previous epoch key to decrypt in-flight messages encrypted under the old epoch
6. All devices rotate independently based on their own elapsed time -- the epoch number in the wire format ensures they use the correct key regardless of minor clock drift

### Clock Synchronization

Epoch rotation does **not** require precise clock synchronization between devices. Each device:

- Tracks elapsed time since it loaded the PSK
- Advances epochs based on elapsed time, not wall clock
- Embeds the epoch number in every encrypted message
- Derives the correct decryption key from the embedded epoch number

Devices that have been offline will derive the necessary key on-demand when they receive a message with a higher epoch number.

## Wire Format

### V1 (No Epoch Rotation)

```
Offset  Length  Field
0       1       Marker (0xFE)
1       1       Version (0x01)
2       12      IV (random nonce)
14      N+16    Ciphertext + GCM Authentication Tag (16 bytes)
```

Total overhead: **30 bytes** (marker + version + IV + auth tag)

### V2 (Epoch Rotation Enabled)

```
Offset  Length  Field
0       1       Marker (0xFE)
1       1       Version (0x02)
2       4       Epoch number (big-endian unsigned)
6       12      IV (random nonce)
18      N+16    Ciphertext + GCM Authentication Tag (16 bytes)
```

Total overhead: **34 bytes** (marker + version + epoch + IV + auth tag)

The `0xFE` marker byte is chosen to not collide with:
- Valid protobuf field tags (`0x08`, `0x10`, etc.)
- Zlib headers (`0x78`)
- Legacy encryption marker (`0xEE`)
- Codec2 audio marker (`0xC2`)

## Packet Size Budget

Meshtastic LoRa packets have a **231-byte** maximum payload (conservative). With encryption overhead:

| Message Type | Typical Size | + V1 (30B) | + V2 (34B) | Fits Single Packet? |
|---|---|---|---|---|
| PLI | 80-120 bytes | 110-150 bytes | 114-154 bytes | Yes |
| Chat (short) | 100-180 bytes | 130-210 bytes | 134-214 bytes | Usually |
| Chat (long) | 200+ bytes | 230+ bytes | 234+ bytes | Fountain coded |
| Generic CoT | Variable | Variable | Variable | Fountain coded if >231 |

Maximum plaintext that fits in a single LoRa packet:
- V1: 231 - 30 = **201 bytes**
- V2: 231 - 34 = **197 bytes**

Messages exceeding 231 bytes after encryption are automatically sent using fountain coding (Luby Transform codes), which provides reliable delivery over lossy LoRa links.

## Interoperability

- **Encrypted device to non-encrypted device**: The non-encrypted device receives a `0xFE`-prefixed payload that fails protobuf parsing. The message is silently dropped.
- **Non-encrypted device to encrypted device**: The encrypted device receives a valid protobuf (no `0xFE` marker) and processes it normally. Unencrypted messages are accepted when encryption is enabled.
- **Mismatched PSKs**: The receiving device attempts decryption, the GCM authentication tag check fails, and the message is silently dropped.
- **Legacy encryption (0xEE)**: The legacy `CryptoUtils` encryption (marker `0xEE`) and the new app-layer encryption (marker `0xFE`) coexist without conflict. The receive path checks for both markers.

## Device Compromise Response

If a device is compromised (lost, stolen, or suspected of key extraction):

1. **Immediately** generate a new PSK: `openssl rand -base64 32`
2. Distribute the new PSK to all remaining team members via a secure channel
3. All team members enter the new PSK (this resets the epoch counter to 0)
4. The compromised device's old key can no longer decrypt new traffic
5. If epoch rotation was active, the attacker cannot decrypt messages from previous epochs (forward secrecy)
6. If the compromised device had TAK Server access, revoke its certificate on the TAK Server

**If epoch rotation was NOT active:** The attacker can decrypt all past traffic captured with the compromised key. This is why epoch rotation is recommended for operational deployments.

## Security Considerations

### Cryptographic Primitives

- **Cipher**: AES-256-GCM (NIST SP 800-38D)
- **IV**: 12-byte (96-bit) random nonce generated by `SecureRandom` for each message
- **Authentication tag**: 128 bits (full GCM tag length)
- **Key derivation from PSK**: SHA-256 (single pass, no salt or iteration count)
- **Epoch key derivation**: HMAC-SHA256 forward chain

### Known Limitations

- **No key stretching**: The PSK is hashed with a single SHA-256 pass. If operators use low-entropy passphrases (against the documented guidance), brute-force attacks are feasible. Always use `openssl rand -base64 32` for key generation.
- **No replay protection**: An attacker who captures encrypted LoRa packets can retransmit them. For PLI data this is low severity since stale positions are overwritten. For chat messages, replayed messages would appear as duplicates.
- **No sender authentication**: Any device with the PSK can send messages appearing to be from any callsign. The PSK model provides group authentication only.
- **Key storage**: The PSK is stored in Android SharedPreferences (plaintext XML on disk). On a rooted device, this file is readable. The `storeKeyInKeystore()` API exists for hardware-backed storage but requires explicit integration.
- **Key material in memory**: Java `String` objects holding the PSK cannot be reliably zeroed due to immutability. `SecretKeySpec` internal byte arrays cannot be zeroed due to JCA design. Key references are set to `null` on cleanup, but garbage collection timing is not guaranteed.
